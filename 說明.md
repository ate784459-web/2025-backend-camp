# 專案環境變數與版本控制說明

## .env.example 的作用

`.env.example` 是一個**環境變數範例檔案**，其主要作用為：

1. **提供設定模板**：展示專案所需的所有環境變數及其預期格式
2. **快速設定參考**：開發者可以快速了解需要配置哪些變數
3. **版本控制友好**：通常會被提交到 Git，而實際的 `.env` 檔案（包含敏感資訊）則會被忽略
4. **團隊協作**：新開發者加入時，可複製此檔案並填入自己的設定值

### .env.example 包含的設定內容

- **PostgreSQL 設定**：用戶名、密碼、資料庫名稱
- **資料庫連接設定**：主機、port、使用者名稱等
- **前端 API 連結**：Vue 前端的 API 基礎 URL
- **後端服務設定**：JWT 密鑰、日誌級別、Firebase 配置等

### 使用方式

1. 複製此檔案為 `.env`
2. 根據開發環境修改實際的值
3. 將 `.env` 加入 `.gitignore` 避免洩露敏感資訊

---

## .env 檔案上傳到 GitHub

### 為什麼 .env 不會被上傳？

`.env` 檔案**不會被上傳到 GitHub**，原因是：

1. **安全考量**：`.env` 包含敏感資訊（JWT 密鑰、資料庫密碼、API 金鑰等），不應該公開在版本庫中
2. **被 .gitignore 忽略**：專案通常會在 `.gitignore` 中加入 `.env`，讓 Git 自動忽略此檔案

### 多開發者協作流程

```
開發者 A：
  複製 .env.example → 建立自己的 .env → 在本地使用

開發者 B：
  複製 .env.example → 建立自己的 .env → 在本地使用

上傳到 GitHub：
  只有 .env.example 被上傳，.env 被忽略
```

這樣每個開發者和部署環境都可以有自己的環境變數設定，而不會洩露敏感資訊。

---

## 不小心上傳 .env 的情況

### 常見的上傳情況

1. **忘記設定 .gitignore**
   - 新專案建立時沒有正確配置
   - `.env` 就會被上傳

2. **先上傳後設定 .gitignore**
   - 不小心先 `git add .env` 並 commit
   - 後來才加入 .gitignore
   - 但敏感資訊已在 Git 歷史中（很難完全刪除）

3. **多個 `.env` 變體沒全部忽略**
   - 如 `.env.local`, `.env.production` 等都需要加入 .gitignore

4. **IDE 或工具自動提交**
   - 一些開發工具可能自動添加檔案

### 防止方法

在 `.gitignore` 中應該包含：

```
.env
.env.local
.env.*.local
*.env
*/*.env*
```

### 如果已經不小心上傳

```powershell
# 從版本歷史中刪除（但只是隱藏，不會真正刪除）
git rm --cached .env
git commit -m "Remove .env file"

# 更安全的做法：使用 git filter-branch 或 BFG Repo-Cleaner
# 但這需要小心操作，可能影響團隊
```

**重點**：如果敏感資訊已經被公開上傳，應該立即更換所有密鑰（JWT、資料庫密碼等）！

## .gitignore 的作用

`.gitignore` 就是用來**告訴 Git 忽略哪些檔案**，寫在裡面的東西 Git 會自動跳過，不會上傳到版本庫。

### 本專案 .gitignore 的設定

此專案中已正確設定：

```
.env        # 忽略 .env 檔案
.env.test   # 忽略 .env.test 檔案
*.env       # 忽略所有 .env 開頭的檔案
*/*.env*    # 忽略所有子目錄中的 .env 相關檔案
```

其他被忽略的內容：

- `node_modules/` → 忽略 npm 套件（很大，不用上傳）
- `logs/`, `*.log` → 忽略所有日誌檔案
- `coverage/` → 忽略測試覆蓋率報告
- `.nyc_output/` → 忽略測試輸出
- 等等...

### 結論

✅ 在這個專案中，`.env` 檔案**不會被上傳到 GitHub**，開發者在本地建立的 `.env` 也會被自動忽略。

## Node.js 與 npm 的作用

### Node.js 是什麼？

**Node.js** 是一個 **JavaScript 執行環境**，讓你可以在伺服器端執行 JavaScript 程式碼（不只在瀏覽器中）。

#### 為什麼需要 Node.js？

1. **後端開發**：用 JavaScript 編寫伺服器邏輯（API、資料庫操作等）
2. **跨平台**：在 Windows、Mac、Linux 都能執行
3. **高效能**：基於事件驅動，適合 I/O 密集型應用
4. **生態豐富**：有龐大的套件庫（npm）

### npm 是什麼？

**npm** (Node Package Manager) 是 **Node.js 的套件管理器**，用來：

1. **安裝套件**：執行 npm install 下載需要的函式庫
2. **管理依賴**：追蹤專案需要哪些套件及版本
3. **執行命令**：透過 npm scripts 執行快捷命令

### Node.js 與 npm 的關係

安裝 Node.js ↓ 自動附帶 npm ↓ 用 npm 安裝套件 ↓ 在 Node.js 環境中執行程式

**簡單比喻**：

- Node.js = 汽車引擎（執行程式的工具）
- npm = 加油站（提供套件/燃料的地方）

## package.json 的用意

`package.json` 是 **Node.js 專案的配置檔案**，裡面記錄**專案的所有配置資訊**。

### package.json 的內容說明

{
"name": "2025-backend-camp", // 專案名稱
"version": "1.0.0", // 專案版本號
"description": "", // 專案描述
"main": "index.js", // 主要進入點檔案

"scripts": {
"start": "docker compose ...", // npm start 時執行的命令
"restart": "docker compose ...", // npm restart 時執行的命令
"stop": "docker compose ...", // npm stop 時執行的命令
"clean": "docker compose ..." // npm clean 時執行的命令
},

"repository": {
"type": "git", // 版本控制系統類型
"url": "git+https://github.com/..." // GitHub 倉庫網址
},

"author": "", // 作者名稱
"license": "ISC", // 開源授權方式

"bugs": {
"url": "https://github.com/..." // 回報 Bug 的網址
},

"homepage": "https://github.com/..." // 專案首頁
}

### 三個最重要的部分

| 部分                | 作用                                         |
| ------------------- | -------------------------------------------- |
| **scripts**         | 定義快捷命令，如 npm start                   |
| **dependencies**    | 列出專案運行時需要的套件                     |
| **devDependencies** | 列出開發時才需要的套件，如測試工具、編譯器等 |

### package.json 的使用流程

1. 開發者複製專案
2. 執行 npm install
3. npm 讀取 package.json 的 dependencies
4. 自動下載安裝所有需要的套件
5. 執行 npm start 啟動專案

### 總結

| 項目             | 說明                                       |
| ---------------- | ------------------------------------------ |
| **Node.js**      | JavaScript 執行環境（像是 Java 的 JVM）    |
| **npm**          | Node.js 的套件管理器（像是 Python 的 pip） |
| **package.json** | 專案配置檔（記錄依賴和命令）               |
| **關係**         | npm 需要 Node.js 才能執行                  |

**package.json 就像是專案的「配方」**，告訴 npm：「要裝什麼東西、怎麼啟動」！

## package.json 中執行 Docker 的原理

### npm scripts 可以執行任何系統命令

`package.json` 中的 `scripts` **不只能執行 Node.js 命令**，它可以執行**任何系統命令**。

```json
"scripts": {
  "start": "docker compose --env-file .env up -d --build"
}
```

### 執行流程

```
npm start
    ↓
npm 讀取 package.json 的 start script
    ↓
npm 把命令傳給作業系統
    ↓
作業系統執行: docker compose --env-file .env up -d --build
    ↓
docker compose 工具讀取 docker-compose.yml
    ↓
啟動容器
```

### npm 是一個中介

```
你 → npm → 作業系統 → docker compose
```

npm 的角色是：

- 讀取 `package.json`
- 把命令轉發給作業系統
- 不涉及 Node.js 本身

### docker-compose.yml 不是被「執行」

准確地說：

- ❌ 執行 docker-compose.yml 檔案
- ✅ 執行 `docker compose` 命令，這個命令會**讀取** docker-compose.yml

## npm scripts 的安全性

### npm scripts 執行需要你同意

你寫在 `package.json` 中的命令**不會自動執行**：

```json
"scripts": {
  "start": "docker compose..."
}
```

**必須由你主動執行**：

```powershell
npm start      # 你親手輸入這個
```

npm **不會自動執行**，需要你明確同意。✅

### ⚠️ 但有一個安全風險：postinstall 腳本

當你執行 `npm install` 時，安裝的套件可能有 **`postinstall` 腳本**會自動執行：

```json
// 某個套件裡面的 package.json
{
  "scripts": {
    "postinstall": "惡意程式碼" // 自動執行！
  }
}
```

**流程**：

```
npm install
    ↓
下載套件
    ↓
發現有 postinstall 腳本
    ↓
自動執行它（不問你同意！）⚠️
```

### 如何防護？

1. **檢查 package.json**：安裝前看一下，確認沒有可疑的腳本
2. **使用 npm audit**：`npm audit` 檢查已知的安全漏洞
3. **從可信來源安裝**：只從官方或知名的 npm 套件安裝
4. **限制自動腳本**：`npm install --ignore-scripts` 安裝時不執行任何腳本

### 安全性總結

| 狀況                 | 執行方式     | 需要同意？          |
| -------------------- | ------------ | ------------------- |
| **npm start**        | 你手動輸入   | ✅ 需要（你主動）   |
| **npm install**      | 你手動輸入   | ✅ 需要（你主動）   |
| **postinstall 腳本** | npm 自動執行 | ❌ 不問（自動執行） |

## 為什麼要通過 npm 執行系統命令？

### 其實不一定要通過 npm！

你可以直接在 PowerShell 中執行：

```powershell
docker compose --env-file .env up -d --build
```

完全不需要 npm！✅

### npm scripts 的真正目的

npm 不是「必須的中介」，而是**為了方便**：

#### 不用 npm：

```powershell
docker compose --env-file .env up -d --build      # 很長，難記
docker compose --env-file .env up --force-recreate --build -d   # 更長
docker compose --env-file .env stop               # 又要記一個
docker compose --env-file .env down -v            # 又要記一個
```

#### 用 npm：

```powershell
npm start      # 簡潔
npm restart    # 簡潔
npm stop       # 簡潔
npm clean      # 簡潔
```

### npm scripts 的優勢

1. ✅ **統一命令**：團隊都用 `npm start`，不用記複雜命令
2. ✅ **易於維護**：改命令只需改 package.json，不用改每個開發者的習慣
3. ✅ **文件化**：把常用命令寫在這裡，新人一看就懂
4. ✅ **跨平台**：Windows、Mac、Linux 都能用一樣的 npm 命令

### 與 Node.js 環境無關

**不是因為在 Node.js 環境才用 npm**。npm 可以執行任何系統命令，和 Node.js 沒有必然關係。

你也可以在非 Node.js 專案中使用 npm scripts，只要安裝了 Node.js。

### 執行流程總結

```
npm start（你主動輸入）
    ↓
npm 協調 → 作業系統執行命令
    ↓
命令讀取設定檔（docker-compose.yml）
    ↓
啟動內容（容器）
```

**npm 是讓開發更方便的工具，不是「必須的」。** 它和「在不在 Node.js 環境」無關，純粹是為了提高開發效率。🎯

## docker-compose.yml 的用意

`docker-compose.yml` 是 **Docker 容器編排檔案**，用來**定義和管理多個容器的協作**。

### 檔案的用意

告訴 Docker Compose：

- ✅ 需要啟動哪些服務（容器）
- ✅ 每個服務如何配置
- ✅ 服務之間如何通訊
- ✅ 資料如何儲存

### 這個專案中的三個服務

#### 1️⃣ **postgres**（資料庫）

啟動 PostgreSQL 資料庫，定義用戶名、密碼、資料庫名稱，設定資料持久化存儲。

#### 2️⃣ **backend**（後端 Node.js）

啟動 Node.js 後端 API 伺服器，使用 ./backend/Dockerfile 構建，等待資料庫健康才啟動。

#### 3️⃣ **frontend**（前端 Vue）

啟動 Vue 前端應用，使用 ./frontend/Dockerfile 構建，將 API URL 傳入。

### 關鍵概念

| 概念            | 說明                                            |
| --------------- | ----------------------------------------------- |
| **services**    | 定義要啟動的容器（postgres、backend、frontend） |
| **volumes**     | 資料持久化（資料庫資料不會因為容器關閉而消失）  |
| **networks**    | 讓容器能互相通訊（bootcamp_network）            |
| **depends_on**  | 定義啟動順序（backend 等待 postgres 健康）      |
| **healthcheck** | 檢查服務是否正常運行                            |
| **env_file**    | 使用 .env 檔案中的環境變數                      |

### 簡單比喻

docker-compose.yml = 樂團指揮指揮譜

三個容器 = 三個樂手，docker-compose 的作用：
✅ 規定三個樂手各自彈什麼
✅ 規定他們什麼時候開始彈
✅ 規定他們怎麼協調配合
✅ 指揮讓他們同時演奏

**所以 docker-compose.yml 就是整個專案的「編排樂譜」**！ 🎼

## docker-compose.yml 不是被執行

### 不是執行檔案

```
❌ 作業系統執行 docker-compose.yml
✅ 作業系統執行 docker compose 命令，該命令讀取 docker-compose.yml
```

### 與 .env 的對比

| 檔案                   | 是否被執行 | 作用                         |
| ---------------------- | ---------- | ---------------------------- |
| **.env**               | ❌ 不執行  | **環境變數配置檔**           |
| **docker-compose.yml** | ❌ 不執行  | **容器編排配置檔**           |
| **npm script**         | ✅ 執行    | **執行 docker compose 命令** |

**所以 docker-compose.yml 是「設定檔」，不是「可執行檔」。** 它被 docker compose 命令讀取，但本身不被執行。✅

## 如何學習編寫 package.json

### 1️⃣ **npm 官方文檔（最重要）**

**npm 官方文檔**：https://docs.npmjs.com/

包含 package.json 完整參考、每個欄位的詳細說明、常見配置和最佳實踐。

**推薦閱讀**：

```
https://docs.npmjs.com/cli/v10/configuring-npm/package-json
```

### 2️⃣ **常用的 package.json 欄位**

| 欄位                | 用途         | 例子                     |
| ------------------- | ------------ | ------------------------ |
| **name**            | 專案名稱     | `"my-project"`           |
| **version**         | 版本號       | `"1.0.0"`                |
| **scripts**         | 快捷命令     | `"start": "node app.js"` |
| **dependencies**    | 生產環境依賴 | `"express": "^4.18.0"`   |
| **devDependencies** | 開發環境依賴 | `"jest": "^29.0.0"`      |

### 3️⃣ **dependencies vs devDependencies**

#### dependencies（生產環境）

```json
"dependencies": {
  "express": "^4.18.0",      // API 框架
  "postgres": "^14.0.0"      // 資料庫驅動
}
```

**這些是執行程式時需要的**。

#### devDependencies（開發環境）

```json
"devDependencies": {
  "jest": "^29.0.0",         // 測試工具
  "nodemon": "^3.0.0",       // 自動重啟工具
  "eslint": "^8.0.0"         // 程式碼檢查工具
}
```

**這些只在開發時用，部署時不需要**。

### 4️⃣ **scripts 的常見命令**

```json
"scripts": {
  "start": "node app.js",              // 啟動應用
  "dev": "nodemon app.js",             // 開發模式
```

# 為什麼一定要用JavaScript來開發網站的前後端？

## 背景：技術選擇的權衡

這個專案選擇了 **JavaScript (Node.js + Vue.js)** 作為技術棧。這不是「非用不可」，而是一個**權衡決策**。

### JavaScript 的優勢

#### 1. **統一語言**

- 前後端都用 JavaScript
- 開發者可以全棧開發，減少學習成本
- 程式碼邏輯可以在前後端複用（如驗證規則）

#### 2. **開發效率高**

- Node.js 生態豐富（npm 有數百萬個套件）
- 框架成熟：Express（後端）、Vue（前端）
- 快速原型開發，適合創業和中小型專案

#### 3. **學習曲線溫和**

- JavaScript 是最容易學的程式語言之一
- 瀏覽器直接執行，容易驗證
- 社群資源豐富，解決問題容易

#### 4. **即時開發體驗**

- Hot reload：修改程式碼無需重啟
- 快速反饋迴圈，提高開發速度
- 適合敏捷開發流程

### JavaScript 的劣勢

#### 1. **性能相對較弱**

- 對比 Java、Go、Rust：執行速度慢
- 不適合計算密集型應用（AI、數據分析）
- 高併發情況下性能有限制

#### 2. **類型系統較弱**

- 動態類型容易引入 bug
- 大型專案需要 TypeScript 補足（增加複雜度）
- IDE 提示支援不如靜態類型語言

#### 3. **缺乏強制的架構規範**

- 開發者容易寫出「義大利麵程式碼」
- 代碼庫管理需要紀律
- 大型團隊需要更多代碼審查

#### 4. **內存管理沒有細粒度控制**

- 不適合嵌入式系統或硬體相關開發
- 無法最佳化特殊情況下的性能

### 其他技術棧的對比

| 技術棧         | 優勢                       | 劣勢                   | 適合場景               |
| -------------- | -------------------------- | ---------------------- | ---------------------- |
| **JavaScript** | 開發快、學習簡單、全棧統一 | 性能一般、類型弱       | 創業、MVP、中小型專案  |
| **Java**       | 性能好、類型強、企業級成熟 | 開發慢、部署複雜       | 大型企業應用           |
| **Go**         | 性能優秀、並發強、部署簡單 | 生態相對較小、學習陡峭 | 微服務、高性能後端     |
| **Python**     | 開發快、數據科學友善       | 性能差、部署複雜       | 數據分析、ML、原型開發 |
| **Rust**       | 性能最佳、內存安全、並發強 | 學習陡峭、開發複雜     | 系統軟體、高性能應用   |

### 為什麼這個專案選 JavaScript？

根據專案特點分析：

#### ✅ **適合用 JavaScript 的原因**

1. **專案規模**：中型專案（健身課程預訂系統）
   - 不是企業級複雜系統
   - 也不是簡單靜態網站
   - JavaScript 正好適合

2. **開發效率優先**
   - 創業/初期開發階段重視快速迭代
   - 需要快速驗證商業模式
   - JavaScript 最合適

3. **業務邏輯相對簡單**
   - 主要是 CRUD 操作（Create、Read、Update、Delete）
   - 不涉及複雜計算或高併發
   - JavaScript 完全勝任

4. **團隊背景**
   - Web 開發背景的團隊
   - 已有 JavaScript 經驗
   - 無需重新學習新語言

### 如果用其他語言會怎樣？

#### 用 Java

```
優勢：性能穩定、大規模擴展性好
劣勢：開發速度慢 50%、部署複雜、學習成本高
適合：銀行、金融、大型電商（併發量大）
❌ 這個專案不需要
```

#### 用 Go

```
優勢：性能好、並發處理強、部署簡單
劣勢：Web 框架沒 Node.js 成熟、生態小
適合：微服務、API Gateway、高併發服務
⚠️ 可以用，但過度設計
```

#### 用 Python

```
優勢：開發快、代碼簡潔
劣勢：性能差、不能前後端統一、部署複雜
適合：數據分析、機器學習、後端 API
❌ 前後端分離困難
```

### 💡 結論

**JavaScript 不是「非用不可」，而是「最合適的選擇」**

對於這個專案：

- ✅ 開發效率優先 → JavaScript 最快
- ✅ 業務邏輯簡單 → JavaScript 足夠
- ✅ 學習成本低 → JavaScript 最易上手
- ✅ 全棧統一 → JavaScript 獨有優勢

**「最好的技術棧是適合當前需求的技術棧」** — 沒有絕對的「最好」，只有「最合適」。

如果這個專案未來：

- 用戶量暴增，需要優化性能 → 可能要用 Go 重寫後端
- 業務變複雜，需要強類型 → 可以轉用 TypeScript + Node.js
- 進入企業級，需要穩定性 → 可能要用 Java 重構

但現階段，JavaScript 是最務實的選擇。✅

# 那Vue.js是程式語言嗎?

## 問題澄清

**不是！Vue.js 不是程式語言。**

許多初學者容易混淆這些概念。讓我們系統地區分它們。

## 四個重要概念

### 1️⃣ **程式語言 (Programming Language)**

**定義**：與電腦溝通的規則和語法

#### 常見的程式語言

```
JavaScript    ← 你正在用的語言
Java          ← 企業應用常用
Python        ← 數據分析常用
Go            ← 微服務常用
Rust          ← 系統軟體常用
C++           ← 遊戲引擎常用
```

**特點**：

- 有官方規範（如 ECMAScript 規範）
- 語法固定（if、for、function 等）
- 獨立存在，不依賴任何框架

### 2️⃣ **執行環境 (Runtime Environment)**

**定義**：讓程式語言程式碼能執行的軟體

#### 常見的執行環境

```
Node.js       ← JavaScript 在伺服器端的執行環境
Java Runtime  ← Java 的執行環境
Python 3.x    ← Python 的執行環境
.NET Runtime  ← C# 的執行環境
```

**特點**：

- 包含編譯器/直譯器
- 提供系統 API 訪問
- 管理記憶體和程序生命週期

**JavaScript 和 Node.js 的關係**：

```
JavaScript = 語言
Node.js = 執行環境（讓 JavaScript 在伺服器運行）
關係：在 Node.js 中執行 JavaScript 程式碼
```

### 3️⃣ **框架 (Framework)**

**定義**：加快開發的預先構建好的代碼庫和工具集

#### 常見的框架

**前端框架**：

```
Vue.js        ← 這個專案用的前端框架
React         ← Facebook 開發的前端框架
Angular       ← Google 開發的前端框架
Svelte        ← 輕量級前端框架
```

**後端框架**：

```
Express.js    ← Node.js 的後端框架（這個專案用的）
Django        ← Python 的後端框架
Spring Boot   ← Java 的後端框架
Gin           ← Go 的後端框架
```

**特點**：

- 基於某種程式語言
- 提供標準化的開發方式
- 簡化常見任務（路由、資料庫連接等）
- 可以被替換（同一語言可用不同框架）

### 4️⃣ **工具 (Tool)**

**定義**：幫助開發的輔助軟體

```
npm           ← 套件管理工具
Docker        ← 容器化工具
Git           ← 版本控制工具
VS Code       ← 編輯器
webpack       ← 打包工具
```

## 四者之間的關係

```
JavaScript（語言）
    ↓
編寫程式碼（語法規則）
    ↓
Node.js（執行環境）或 瀏覽器（執行環境）
    ↓
Express.js 或 Vue.js（框架）
    ↓
npm、Docker 等（工具）
    ↓
完成開發
```

## 這個專案中的各層次

### **後端層次**

```
1. 程式語言        → JavaScript
2. 執行環境        → Node.js
3. 後端框架        → Express.js
4. 套件管理工具    → npm
5. 容器化工具      → Docker
```

### **前端層次**

```
1. 程式語言        → JavaScript
2. 執行環境        → 瀏覽器（Chrome、Firefox 等）
3. 前端框架        → Vue.js
4. 套件管理工具    → npm
5. 打包工具        → Vite
```

### 2️⃣ **「它有語法規則（if、for、function）嗎？」**

- 是 → 很可能是程式語言
- 否 → 很可能是框架或工具

### 3️⃣ **「它依賴某個程式語言嗎？」**

- 是 → 很可能是框架或工具
- 否 → 很可能是程式語言或執行環境

**記住這個核心關係**：
